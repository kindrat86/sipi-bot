Spend Controls for OpenAI Agents SDK

The OpenAI Agents SDK gives your agent tools, handoffs, and prompt-level guardrails — but no built-in spend controls. Add sipi.bot as a tool and every payment function call gets evaluated against your rules before it executes: per-transaction caps, daily totals, velocity limits, and merchant allowlists.

The OpenAI Agents SDK gap. Agents built with the SDK can call any tool you give them — including tools that trigger real payments. A handoff chain where each step calls a paid API, or a retry loop on a flaky tool, can rack up significant spend before anyone notices. The SDK has no concept of "should this spend be allowed." sipi.bot adds that check as one tool call before the paid action fires.

What you get

  • Pre-spend evaluation — every payment tool call is checked against your rules in under 5ms, before it executes.
  • Per-transaction caps — block any single spend over a ceiling you set.
  • Daily totals — rolling budget across all agent transactions.
  • Velocity limits — kill retry loops and runaway handoff chains on the Nth repeat.
  • Merchant allowlists — block the agent from spending at any vendor you haven't approved.
  • Human-in-the-loop approval queue — large or ambiguous spend is FLAGGED and held for a human.
  • Tamper-evident audit log — every decision, immutable, with the rule that fired.

The integration: 15 lines

Register sipi.bot as a tool, and have every payment function check with it first. On BLOCKED, abort. On FLAGGED, route to a human.

from agents import Agent, function_tool
import requests

@function_tool
def evaluate_spend(amount: float, merchant: str, category: str) -> dict:
    """Check whether this spend is allowed before paying."""
    r = requests.post(
        "https://sipi.bot/v1/transactions/evaluate",
        headers={"Authorization": f"Bearer {SIPBOT_KEY}"},
        json={"amount": amount, "merchant": merchant, "category": category},
        timeout=5,
    ).json()
    return r  # {"decision": "APPROVED|BLOCKED|FLAGGED", "reason": "..."}

@function_tool
def buy_compute(amount: float, vendor: str) -> str:
    """Purchase GPU compute — gated by sipi.bot."""
    decision = evaluate_spend(amount, vendor, "compute")
    if decision["decision"] != "APPROVED":
        return f"Blocked: {decision['reason']}"
    # ... your actual purchase logic here
    return "Purchased"

agent = Agent(
    name="procurement-agent",
    instructions="You buy compute for the team. Always check evaluate_spend first.",
    tools=[evaluate_spend, buy_compute],
)

The spend rules that matter most for OpenAI agents

1. Velocity limit (the retry-loop killer)

OpenAI Agents SDK agents that use tool-calling in a loop are the most common source of runaway spend. A tool fails, the agent retries, the retry fails, the agent retries again — each retry is a paid call. A velocity rule caps the number of transactions per window and kills the loop on the Nth attempt.

{"rule_type": "velocity", "params": {"max_count": 10, "window_minutes": 60}, "action": "BLOCKED"}

2. Per-transaction cap

Even with velocity under control, a single catastrophic spend — a $5,000 GPU rental, a misformatted API call that bills at a premium tier — needs a hard ceiling. Block anything over $500 outright.

{"rule_type": "per_transaction", "params": {"max_amount": 500}, "action": "BLOCKED"}

3. Merchant allowlist

Agents that discover novel vendors during tool use (or that have been prompt-injected to route spend to an attacker-controlled merchant) are blocked by default. Only vendors you've approved go through.

{"rule_type": "merchant_allow", "params": {"allowed": ["openai.com", "aws.amazon.com"]}, "action": "BLOCKED"}

Why not just use OpenAI's spending limits?

OpenAI's account-level spend limits are billing controls on your OpenAI account — they cap total spend with OpenAI, not what your agent spends. If your agent buys compute from AWS, advertising from Meta, or data from a third-party API, OpenAI's limit does nothing. sipi.bot governs spend across any payment the agent can trigger, not just calls to OpenAI.
