Velocity Limit

Velocity limit — a spend-firewall rule that caps the number of transactions an AI agent can make in a time window. It is the primary defense against retry loops, the #1 cause of runaway agent spend (44% of incidents).

Definition

A velocity limit is a count-based rule (as opposed to an amount-based rule like a per-transaction cap or daily total). It restricts how many transactions an agent can make per unit of time, regardless of how much each transaction costs. A typical velocity rule: maximum 10 transactions per hour.

{"rule_type": "velocity", "params": {"max_count": 10, "window_minutes": 60}, "action": "BLOCKED"}

Why velocity matters more than amount

Most spend controls are amount-based — a per-transaction cap, a daily total, a monthly budget. These catch the big single spend or the aggregate bleed. But neither catches the pattern that actually causes catastrophic spend: many small transactions in rapid succession.

A retry loop makes 40 calls in 90 seconds. Each call might cost only $0.50 — well under any per-transaction cap. But 40 × $0.50 = $20 in 90 seconds, and if the loop runs for an hour, that's $800. An amount-based daily cap of $1,000 would not trip until $1,000 is already gone. A velocity limit of 10 transactions per hour kills the loop on the 11th attempt, before $6 has been spent.

The math. A velocity rule is the cheapest insurance in spend control. It costs nothing to configure, runs in <5ms, and prevents the single most common catastrophic failure mode. If you configure only one rule, make it a velocity limit.

How it works in sipi.bot

Every transaction the agent attempts is counted in a rolling window. When the count exceeds max_count within window_minutes, the next transaction is BLOCKED with reason "velocity limit exceeded." The agent must wait for the window to slide before it can transact again.

Velocity limits compose with the other rule types — per-transaction caps, daily totals, merchant allowlists, category limits, time windows. The most effective configuration layers velocity (kills loops) with a per-transaction cap (catches the big one) and a merchant allowlist (blocks unknown vendors).
